# Analysis Framework

## Evaluation Brief Structure

For each cluster (and individually for orphans), produce:

```markdown
## [Cluster Theme Name] (IDs: XXX, XXX, XXX)

### Feasibility
- **Effort estimate:** [Low / Medium / High / Very High]
- **Key requirements:** [What skills, tools, or infrastructure does this need?]
- **Bottleneck:** [The single hardest part of delivering this]
- **Existing solutions:** [Does something like this already exist? How close?]

### Impact
- **Audience reach:** [How many community members would this serve?]
- **Value depth:** [Surface convenience vs. fundamentally changes how they work]
- **Urgency:** [Is this a burning need or a nice-to-have?]
- **Reusability:** [One person's solution or reusable by many?]

### Novelty
- **What exists:** [Current alternatives, even partial ones]
- **Differentiation:** [What would make this version worth building despite alternatives?]
- **Innovation angle:** [Does this introduce a new approach or just repackage existing ones?]

### Channel Alignment
- **Content pillar:** [Which pillar does this map to? Or is it a new pillar?]
- **Content generation:** [Could building/delivering this produce channel content?]
- **Audience reinforcement:** [Does this attract or retain the target audience?]
- **Brand demonstration:** [Does delivering this demonstrate the channel's thesis in action?]

### Research Alignment
- **MWP fit signal:** [Does this look like it could be an MWP workspace? Initial read only — Stage 04 does rigorous classification]
- **Paper relevance:** [Does this advance, demonstrate, or test the MWP/ICM thesis?]
- **Case study potential:** [Could the process of building this become a case study?]

### Risks
- **Delivery risk:** [What could prevent successful delivery?]
- **Scope risk:** [Could this expand beyond what's manageable?]
- **Reputation risk:** [Could a bad delivery hurt community trust?]
- **Opportunity cost:** [What else doesn't get built if this takes priority?]
```

## Evaluation Discipline

**Be concrete, not vague.** "This would be hard" is not useful. "This requires building a custom API integration with a service that has no public API" is useful.

**Be honest about impact.** Not every request from the community is high-impact. Some are niche. That's fine — but don't inflate impact to be nice. Honest evaluation now prevents wasted effort later.

**Separate requester importance from idea importance.** A key community member requesting something low-value doesn't make the idea high-value. Note the requester context in the evaluation (Stage 04 can weight it), but evaluate the idea on its own merits.

**Flag unknowns.** If you can't assess feasibility because you don't know the technical landscape, say so. "Unknown — requires research into X" is better than guessing.
